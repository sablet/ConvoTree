# 意図のグループ化

## タスク概要

以下の意図リストを分析し、**共通のテーマ・目的を持つ意図をグループ化**してください。

あなたの役割は意図の内容を理解し、どの意図とどの意図が同じグループに属するかを判断することです。インデックス管理やデータ構造の操作は不要です。

## グループ化の基準

{COMMON_INTENT_OBJECT}

**重要**: グループを代表する名前は、上記のintent定義と同様に、志向対象と志向方向性を必ず含むこと。

### 具体性の要求

**CRITICAL**: 抽象的で一般的なグルーピングは禁止です。以下の情報を活用して、**具体的で文脈を反映したグループ名**を作成してください：

1. **source_full_paths**: メッセージの出所（例：「Inbox -> プロジェクトA」）を反映
2. **objective_facts / context**: 具体的な状況や課題を反映
3. **技術・ドメイン**: 特定の技術スタック、ビジネスドメインを明示

**良い例（具体的）**:
- 「チャットアプリのバックエンドAPIの信頼性とパフォーマンスを向上させたい」
- 「データ分析パイプラインにおけるクラスタリング精度を改善したい」
- 「個人の旅行計画における時間管理と意思決定プロセスを最適化したい」

**悪い例（抽象的・一般的）**:
- 「システム・アプリケーションの品質と信頼性を確保したい」（どのシステム？）
- 「個人の生産性とタスク管理能力を向上させたい」（どの文脈？）
- 「データ処理パイプラインを強化したい」（何のデータ？どの領域？）
- 「色々な改善」（志向対象が曖昧）
- 「システム開発」（志向方向性が欠けている）
- 「改善する」（志向対象が欠けている）

## グループ化ルール

1. **具体性優先**: **source_full_paths, objective_facts, context を必ず考慮し、具体的な文脈を反映したグループ名を作成**
2. **テーマの共通性**: 共通のテーマ・目的・領域・**文脈**を持つ意図をグループ化
3. **抽象度の制限**:
   - ❌ 「システム」「アプリケーション」「データ」「開発」のような一般用語のみのグルーピングは禁止
   - ✅ 具体的なシステム名、プロジェクト名、技術名、ドメイン名を含めること
4. **最小1つ以上**: 1つのグループには最低1つ以上の意図が含まれること
5. **重複なし**: 1つの意図は1つのグループにのみ属する（排他的）
6. **網羅性（必須）**: **全ての意図が必ずいずれかのグループに含まれること**
7. **単独意図の扱い**: どのグループにも属さない単独の意図は、それ自体を1つのグループとして扱う

## 出力形式

グループごとに以下の形式で出力:

```json
{{
  "group_name": "グループを代表する意図（志向対象+志向方向性を含む、具体的な文脈を反映）",
  "description": "このグループの詳細説明（具体的な文脈、技術、ドメインを含む）",
  "rationale": "なぜこれらをグループ化したか（共通するsource_full_paths, objective_facts, contextを根拠に）",
  "member_indices": [0, 2, 5]  // このグループに属する意図のインデックス（0始まり）
}}
```

**重要**:
- 出力は純粋なJSON配列のみ（説明文・コメント禁止）
- 全ての意図（0〜{max_index}）がいずれかのグループに含まれること
- 1つの意図が複数のグループに属してはならない
- **group_name は必ず具体的な文脈を含むこと**（「システム」「データ」などの一般用語のみは禁止）

例:
```json
[
  {{
    "group_name": "...",
    "description": "...",
    "rationale": "...",
    "member_indices": [0, 1, 3]
  }},
  {{
    "group_name": "...",
    "description": "...",
    "rationale": "...",
    "member_indices": [2]
  }},
  {{
    "group_name": "...",
    "description": "...",
    "rationale": "...",
    "member_indices": [4, 5]
  }}
]
```
